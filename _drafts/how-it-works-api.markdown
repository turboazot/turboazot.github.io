Черновик для цикла «как устроен IT».

Пользователь нажимает кнопки. Программа вызывает API.

API - одно из тех слов, которые постоянно звучат в IT, но часто объясняются слишком сухо: "интерфейс программирования приложений". Формально верно, но легче от этого не становится.

Проще думать об API как о меню: вот что можно попросить, вот как правильно попросить, вот какой ответ можно получить.

# Аналогия

Представим ресторан.

Ты не заходишь на кухню и не роешься в холодильнике. Ты смотришь меню и делаешь заказ официанту. В меню написано, что можно заказать. У каждого блюда есть название, цена и иногда опции.

API работает похоже. Одна программа не должна лезть внутрь другой. Она делает запрос по правилам, а другая программа отвечает.

# Зачем программам разговаривать

Тезисы для раскрытия:

- сайт просит сервер войти в аккаунт;
- приложение просит карту построить маршрут;
- магазин просит платежную систему списать деньги;
- сервис доставки просит ресторан подтвердить заказ;
- AI-продукт может просить модель сгенерировать ответ.

Главная мысль: современный IT состоит из программ, которые постоянно договариваются друг с другом.

# Меню возможностей

Тезисы:

- API описывает, какие действия доступны;
- например: создать заказ, получить список товаров, обновить профиль;
- у каждого действия есть правила входа;
- есть ожидаемый ответ;
- есть ошибки, если запрос неправильный или действие нельзя выполнить.

Пример:

```text
попросить: создай заказ
передать: товары, адрес, способ оплаты
получить: номер заказа или ошибку
```

# Почему нельзя просто дать доступ ко всему

Тезисы:

- внутренняя кухня программы может быть сложной и опасной;
- API ограничивает, что можно делать снаружи;
- это помогает безопасности;
- это дает стабильность: внутренности можно менять, если API остается прежним;
- хорошее API похоже на нормальную стойку приема заказов, а не на проход на склад.

# Плохой API

Тезисы:

- непонятно, что передавать;
- ошибки ничего не объясняют;
- одно действие делает слишком много;
- названия сбивают с толку;
- поведение меняется без предупреждения;
- программисты начинают угадывать вместо того, чтобы спокойно пользоваться.

Можно привести бытовой пример меню, где "комбо 3" каждый день значит разное.

# Вывод

API - это договор между программами. Он говорит: что можно попросить, как именно попросить и какой ответ ждать. Чем понятнее договор, тем проще разным частям IT-системы жить вместе.

# Куда дальше

Когда программ, данных и API становится много, обязательно что-то ломается. Поэтому дальше - почему программы ломаются и почему баги не всегда означают, что кто-то просто плохо старался.
